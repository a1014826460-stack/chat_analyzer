# 下注统计核心逻辑

本文记录当前版本里，聊天消息从读取、分类到进入下注统计和“机器人汇总校验”的核心规则。

重点说明两件事：

1. 什么格式的聊天消息会被归类成什么“文本类型”。
2. 哪些消息只用于展示，哪些消息会进入统计，哪些消息会进入“机器人汇总校验”。

本文内容基于当前实现，主要对应：

- [app/ui/raw_chat_dialog.py](/d:/pythonProject/outsource/chat_analyzer/app/ui/raw_chat_dialog.py)
- [app/services/chat_service.py](/d:/pythonProject/outsource/chat_analyzer/app/services/chat_service.py)

## 1. 数据读取入口

消息读取入口在 `MainWindowDataMixin._load_filtered_messages()`，最终会调用 `ChatLogService.load_messages_with_cache()`。

SQLite 读取逻辑在 `ChatLogService.load_messages_from_sqlite()`：

- 主消息库通常是 `msg_0.db`
- 同目录 `im.db` 中的 `groupinfo` 用来把群 ID 映射成真实群名
- `groupmemberinfo` / `userinfo` 会补发言人昵称
- 默认只读取最近 5 分钟消息
- 如果启用了高级时间筛选，则按用户指定时间窗口读取
- `ParseOptions.group_ids` 控制群组过滤

## 2. 系统中的文本类型

原始聊天记录窗口支持以下文本类型：

- `下注归期边界`
- `推断期号边界`
- `机器人回执`
- `机器人汇总`
- `开奖结果播报`
- `机器人状态快照`
- `用户下注`
- `撤单/改注`
- `普通聊天`

当前版本里，原始聊天记录窗口主要实际使用到的是：

- `下注归期边界`
- `机器人回执`
- `机器人汇总`
- `开奖结果播报`
- `机器人状态快照`
- `用户下注`
- `撤单/改注`
- `普通聊天`

说明：

- 同一条消息可以同时属于多个类型。
- 目前最常见的并存组合是 `机器人汇总 / 下注归期边界`。
- “显示类型”主要用于原始聊天记录窗口的筛选和高亮，不等于“是否进入下注统计”。

## 3. 文本类型归类规则

以下规则按当前代码真实行为整理。

### 3.1 机器人回执

满足以下条件时归类为 `机器人回执`：

- 文本同时包含 `下注期数`
- 文本同时包含 `下注内容`

典型格式：

```text
@张三 :
下注期数: 3447001
下注内容:
------------
大单2800(4.28赔率)
13.500(12.0赔率)
------------
余额：3747
```

说明：

- 这类消息是机器人对某个下注人的确认回执。
- 在统计逻辑中，这类消息会被解析成回执下注事件。
- 它不是“机器人汇总”。

### 3.2 机器人汇总

满足以下任一形态，并且能从汇总行里解析出投注内容时，归类为 `机器人汇总`。

#### 形态 A：标准“期下注核对”

典型格式：

```text
3447069期下注核对：
许悦 264509[大双17509 小单9000 小双14000]
包红 45000[大双5000 小单9000 小双9000]
```

或：

```text
第3447069期下注核对
--------------------------
天佑 036192[小3100 大双3188 12T66]
紫蔡 031241[大2500 大双788 大单988 20T72]
```

判定特征：

- `3447069期下注核对：`
- `第3447069期下注核对`
- `期下注核对`

#### 形态 B：带 bracket 头的汇总快照

典型格式：

```text
---[H3447069-005]---
3447069期：

执念 62187[大单1100 大双1200]
鼓励 37132[12.200 18.200]
```

判定特征：

- `---[H3447069-005]---`
- `---[3447069]---`
- 后续存在多行可解析的汇总投注内容

#### 形态 C：封盘边界消息里直接附带汇总明细

典型格式：

```text
已截止本期猜猜
3447069 期
-------封盘线--------
以下投注全部有效

加拿大PC蛋蛋
【得闲-1192520】小单25000|大单25000
【肺子-983031】小双36666|大双36666
```

或：

```text
已截止本期猜猜 期号:3447069
加拿大PC蛋蛋
【坚果-330028】大4200|小单2700|大双4200|13.288
【牛肝-58746】大双1500|小单1500
```

判定特征：

- 有期号
- 有 `已截止本期猜猜` / `封盘线` / `以下投注全部有效`
- 后续有 `【用户-编号】投注|投注|投注` 这样的明细行

说明：

- 这类消息会同时归类为 `机器人汇总` 和 `下注归期边界`
- 这正是“并存”的主要场景

#### 形态 D：无期号但明确是“本期下注列表”

典型格式：

```text
-----本期下注列表-----

加拿大PC蛋蛋
【谢佳-50405】小单1600|大双1600|11.60|12.60|13.60
【量仔-42064】小单2005|小双2005|12.100|18.100
【很久-37084】大双1500|小单500|大1500
```

判定特征：

- 文本包含 `本期下注列表`
- 后续有多行 `【用户-编号】...` 可解析投注内容

说明：

- 这类消息现在会归类为 `机器人汇总`
- 即使用户名不含“机器人”两个字，只要是明显的机器人清单格式，也不会再误归类成 `用户下注`

### 3.3 下注归期边界

满足以下条件时归类为 `下注归期边界`：

- 发送者看起来是机器人
- 文本中包含边界类提示词之一：
  - `开始下注`
  - `下注开始`
  - `截止线`
  - `封盘线`
  - `已截止`

典型格式：

```text
已截止本期猜猜
3447069 期
-------封盘线--------
下注结束
```

说明：

- 如果消息只是边界提示，没有汇总明细，则只归类为 `下注归期边界`
- 如果边界提示里带了投注汇总，则会和 `机器人汇总` 并存

### 3.4 开奖结果播报

满足以下条件时归类为 `开奖结果播报`：

- 同时包含 `彩种`
- 同时包含 `期号`
- 同时包含 `结果` 或 `历史开奖`

典型格式：

```text
彩种：加拿大PC蛋蛋
期号:3447026
结果:2+9+8=19 大 单
```

说明：

- 这类消息用于开奖展示，不算机器人汇总

### 3.5 机器人状态快照

满足以下条件时归类为 `机器人状态快照`：

- 同时包含 `当前积分`
- 同时包含 `本期下注` 或 `本期未下注`

典型格式：

```text
---[第3447193期]---
紫蔡 当前积分：10893
当前盈亏：48532
当前流水：1301590
下注冻结：10832
回粮冻结：0
剩余积分：61
本期下注：[小2400 大单3500 双1866 小双3066]
```

说明：

- 这类消息现在只归类为 `机器人状态快照`
- 不会再误归类成 `机器人汇总`
- 但当同组同一期没有官方群汇总时，统计层可以把这类消息当作“机器人汇总兜底来源”去聚合校验
- 也就是说：
  - 展示层：它是 `机器人状态快照`
  - 校验层：必要时可作为兜底汇总数据来源

### 3.6 用户下注

满足以下条件时归类为 `用户下注`：

- 发送者不被视为机器人
- 文本里出现可识别的下注玩法和金额

当前只统计这 8 个玩法：

- `大单`
- `小单`
- `大双`
- `小双`
- `大`
- `小`
- `单`
- `双`

典型格式：

```text
大100
小双200
大单500
```

说明：

- 只有非机器人消息才会被打上 `用户下注`
- 机器人发送的“本期下注列表”不会再被打上 `用户下注`

### 3.7 撤单/改注

满足以下条件时归类为 `撤单/改注`：

- 文本恰好是 `取消`
- 或文本包含 `已取消`
- 或文本包含 `撤单`

典型格式：

```text
取消
如下订单已取消
撤单
```

### 3.8 普通聊天

不满足以上任何规则时，归类为 `普通聊天`。

另外，赔率宣传类消息会被优先直接归类为 `普通聊天`，避免误判：

典型格式：

```text
大小单双 1.98倍
小单大双 3.69倍
小双大单 4.29倍
```

判定特征：

- 包含 `倍`
- 同时包含 `赔率` / `欢迎` / `大小单双` / `吗` 之一

## 4. 一条消息可以同时属于哪些类型

当前允许并存的典型情况：

- `机器人汇总 / 下注归期边界`

典型示例：

```text
已截止本期猜猜 期号:3447069
加拿大PC蛋蛋
【坚果-330028】大4200|小单2700|大双4200|13.288
```

当前明确不应并存的情况：

- `机器人汇总 / 机器人状态快照`

也就是说，个人状态快照现在不会再显示成：

```text
[机器人汇总 / 机器人状态快照]
```

而应该只显示：

```text
[机器人状态快照]
```

## 5. 哪些消息会进入下注统计

统计逻辑不完全等于展示分类。

### 5.1 会进入下注统计的消息

- 直接群里的用户下注消息
- 机器人回执消息
- 机器人汇总消息

### 5.2 不直接进入下注统计，但会影响期号归属的消息

- 下注归期边界消息

### 5.3 不作为普通下注事件进入统计的消息

- 开奖结果播报
- 机器人状态快照
- 普通聊天
- 赔率宣传

## 6. 机器人汇总校验的数据来源

“机器人汇总校验”同时依赖两侧数据：

- 软件侧：来自 `ChatLogService.extract_bet_visual_data()` 生成的同一批下注行，也就是 `实时统计文本` 和 `玩法下注堆积柱形图` 使用的数据源。
- 机器人侧：来自对应群组、对应期数的 `机器人汇总`。

因此，只有同时拿到“同一群组 + 同一期号”的软件侧汇总和机器人侧汇总时，才会生成该群该期的 `机器人汇总校验结果`。

如果只拿到机器人汇总，但同群同期没有软件侧下注统计，系统不会把软件侧当成 0 去校验，也不会生成该期校验结果。

如果软件侧有同群其它期号的下注统计，这些统计也不会参与当前期号的机器人汇总校验。

### 6.1 软件侧来源

软件侧汇总来自实时统计和柱形图共用的 `visual_rows`：

- 只使用普通下注解析后的统计行，例如直接群用户下注、机器人回执解析出的下注。
- 校验时按 `(群组, 期号)` 再按玩法汇总。
- `source_kind == "summary"` 的机器人汇总兜底行不会作为软件侧校验依据，避免机器人汇总自己校验自己。
- 群屏蔽名单、全局屏蔽名单、余额不足拒单等过滤结果会先作用于 `visual_rows`，再进入软件侧校验汇总。

### 6.2 机器人侧来源

“机器人汇总校验”的机器人侧优先使用官方群汇总消息。

优先顺序如下：

1. 权威官方群汇总
   - `3447069期下注核对`
   - `第3447069期下注核对`
   - `---[H3447069-005]--- + 3447069期：`
   - `已截止本期猜猜 + 以下投注全部有效 + 汇总明细`

2. 兜底个人状态快照
   - 仅当同组同一期没有官方群汇总时使用
   - 会把多个 `当前积分 / 本期下注:[...]` 的机器人状态快照按下注人聚合成当期机器人汇总

额外规则：

- 群屏蔽名单会同时作用于机器人汇总校验
- 被屏蔽的下注人不会参与机器人汇总对账
- 私聊代投、代理行不会进入机器人汇总统计
- 校验记录按 `群组 + 期号` 去重，弹窗支持在同一群组下切换查看不同期数
- 历史记录只保存已经成功生成的校验记录；缺少同群同期软件侧数据的期号不会进入历史记录

### 6.3 本地校验记录与异常报告

每次生成 `summary_check_record` 后，系统会把校验结果保存到本地：

```text
<export_dir>/summary_check/YYYY-MM-DD/
```

如果没有设置导出目录，则保存到程序当前目录下的：

```text
summary_check/YYYY-MM-DD/
```

每个群组每期至少保存一份 JSON 记录：

```text
3447321_鸿业鸿信4.73_4.31贵宾厅_record.json
```

如果任一玩法超过容差并被判定为异常，还会额外保存 Markdown 异常报告：

```text
3447321_鸿业鸿信4.73_4.31贵宾厅_exception.md
```

异常报告会列出异常玩法、软件金额、机器人金额、偏差，并根据当前证据自动给出原因方向：

- 机器人侧金额高于软件侧：可能有机器人汇总中的下注未进入软件侧 rows
- 软件侧金额高于机器人侧：可能有软件识别到的下注未进入机器人核对账单
- 没有同群同期软件侧 rows：无法完成有效对账
- 机器人汇总疑似误入用户下注链路：需要检查消息分类

同一群同一期如果校验内容没有变化，不会重复写入文件，避免实时刷新时反复刷屏。

## 7. 主窗口状态与刷新

### 7.1 窗口大小与分割条

主窗口现在会优先恢复上次关闭前的 UI 状态：

- 窗口大小与位置
- 中间 `QSplitter` 的左右分栏位置

如果本地还没有保存过这些状态，则按当前屏幕可用区域动态计算初始窗口大小，并按比例设置初始左右分栏宽度，而不是固定写死像素值。

### 7.2 可见群组持续保持

左侧“可见群组”不再只依赖当前刷新时刻的 `selected_group_ids`，而是额外保存每个群组 ID 的最后一次勾选状态：

- 当前已勾选的群，下一次 5 秒刷新后仍保持勾选
- 当前未勾选的群，下一次 5 秒刷新后仍保持未勾选
- 某个群临时不在本轮列表里，后续重新出现时，会恢复它上次的勾选状态

这保证了“可见群组”筛选不会因为每期刷新而被重置。

### 7.3 5 秒自动刷新减卡顿

5 秒自动刷新仍然会继续拉取最新消息，但 UI 侧现在会先比较本次结果与当前已显示结果是否真的发生变化。

如果以下内容没有变化：

- 当前消息集合
- 当前可视下注行
- 当前统计 totals

则会跳过不必要的 UI 重刷，包括：

- 原始聊天记录窗口刷新
- 图表数据重灌
- 基于累积行的统计重算

这样可以减少周期性刷新时的界面卡顿，同时不影响真实数据更新。

## 8. 原始聊天记录窗口显示规则

原始聊天记录窗口每条消息头部显示：

```text
时间 | 群组 | 用户名 | 期数
```

说明：

- 最后一列显示的是该消息提取到的期数
- 不再显示 `sender_id`
- 搜索时也会把期数纳入搜索字段

## 9. 近期修复过的关键边界

当前版本已经修复以下误判：

- `极大/极小` 不再误归入 `大/小`
- `ABC/定位双` 不再误归入普通 `双`
- 个人状态快照不再误归类为 `机器人汇总`
- `本期下注列表` 这类机器人清单不再误归类为 `用户下注`
- 带封盘边界的汇总消息支持 `机器人汇总 / 下注归期边界` 并存
- “机器人汇总校验结果”按群保留最近 20 期

## 10. 8 个统计玩法的硬限制

右侧下注记录、实时统计文本、群组汇总统计，只统计以下 8 个下注类别：

- `大单`
- `小单`
- `大双`
- `小双`
- `大`
- `小`
- `单`
- `双`

其他玩法即使在原始消息中出现，也不会进入这 8 类核心统计视图。
